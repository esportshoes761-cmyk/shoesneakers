# Plan para Generar Especificación Técnica de Separación de Paneles de Administración

## Objetivo
Generar una especificación técnica y funcional detallada en castellano para dividir el panel de administración actual en dos paneles: Administrador Marketing (mejorado) y Administrador Emulador (auditoría y control), sin alterar el diseño ni UX actual, incluyendo todos los entregables del 1 al 14.

## Pasos del Plan

1. **Investigación del Estado Actual**
   - Revisar el archivo de requisitos (mmmmmmm.txt) para extraer funciones existentes del panel admin.
   - Asumir funciones basadas en descripción del usuario: gestión de productos, marcas, precios, lotes, imágenes, premios, solicitudes, informes de ventas, inventario, exportaciones, promociones.
   - No hay código fuente en el workspace, por lo que la especificación será genérica basada en requisitos.

2. **Estructuración del Documento**
   - Crear documento en castellano con secciones numeradas 1-14.
   - Incluir ejemplos concretos, JSON para esquemas de eventos, pseudo-endpoints.
   - Asignar prioridades (Mínimo viable / Fase 2 / Fase 3) a cada funcionalidad.

3. **Definición de Contenidos por Sección**
   - **1. Resumen ejecutivo**: Párrafo breve sobre separación y beneficios.
   - **2. Roles y permisos**: Listar permisos para Administrador Marketing, Emulador, y roles secundarios.
   - **3. Mapeo de funciones**: Tabla asignando funciones existentes a Marketing o Emulador.
   - **4. Funcionalidades detalladas**: CRUD productos, bulk uploads, dashboards, logs para Marketing; auditoría en tiempo real, reproducción de sesiones, alertas con botones para Emulador.
   - **5. Requisitos de auditoría**: Esquema de eventos, retención, índices, privacidad.
   - **6. API/endpoints**: Nuevos endpoints para logs, alertas, bulk uploads, WebSockets para tiempo real.
   - **7. UI/UX**: Pautas para Marketing conservativo, mockups textuales para Emulador.
   - **8. Seguridad**: 2FA, roles, logging, rate limiting, cifrado.
   - **9. Flujo de despliegue**: Feature flags, A/B testing, rollback.
   - **10. Métricas y monitoreo**: Latencia logs, tamaño almacenamiento, eventos/seg, uso alertas.
   - **11. Casos de uso**: 4 ejemplos concretos.
   - **12. Recomendaciones técnicas**: Bases de datos (Elasticsearch, ClickHouse), message brokers (Kafka), WebSockets, costes.
   - **13. Checklist de pruebas**: Funcionales, seguridad, integridad, rendimiento.
   - **14. Datos mínimos**: Preguntar al usuario por endpoints actuales, esquema BD, volúmenes, retención, regulaciones.

4. **Incorporación de Datos del Usuario**
   - Esperar confirmación de datos mínimos (endpoints, BD, volúmenes, etc.) para personalizar esquemas y ejemplos.
   - Si no se proporcionan, usar placeholders genéricos.

5. **Generación del Documento**
   - Escribir el documento completo basado en el plan.
   - Asegurar ejemplos JSON válidos, pseudo-endpoints con método/ruta/parámetros/respuesta.
   - Mantener lenguaje técnico preciso para desarrolladores.

6. **Validación**
   - Verificar que cubre todos los entregables.
   - Asegurar consistencia entre secciones.

## Riesgos
- Falta de datos específicos del usuario podría hacer la especificación genérica; mitigar preguntando.
- Asunción de funciones actuales basadas en descripción; confirmar con usuario si necesario.

## Timeline
- Día 1: Investigación y estructuración.
- Día 2: Redacción de secciones 1-7.
- Día 3: Redacción de secciones 8-14, incluyendo datos del usuario.
- Día 4: Revisión y finalización.
